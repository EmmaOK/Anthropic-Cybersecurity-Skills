# ASVS v4.0.3 Per-Control Testing Reference

Per-control test steps, commands, and pass/fail criteria for all major testable
requirements in ASVS v4.0.3, organized by chapter. L1 = required for all apps;
L2 = apps handling sensitive data; L3 = critical/high-assurance systems.

Use this alongside the worksheet generated by `agent.py init` to record evidence
against each control ID.

---

## V2 — Authentication Verification

### V2.1.1 (L1) — Minimum password length

**Requirement:** Passwords must be at least 12 characters.

```bash
# Attempt registration with an 11-character password
curl -s -o /dev/null -w "%{http_code}" -X POST $TARGET/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"ShortPass1!"}'
```

**Pass:** HTTP 400 with a message indicating minimum length.  
**Fail:** HTTP 200/201 — account created with a short password.

---

### V2.1.2 (L1) — Maximum password length ≥ 64 characters

```bash
LONG_PASS=$(python3 -c "print('A'*64 + '1!')")
curl -s -o /dev/null -w "%{http_code}" -X POST $TARGET/api/auth/register \
  -H "Content-Type: application/json" \
  -d "{\"email\":\"test2@example.com\",\"password\":\"$LONG_PASS\"}"
```

**Pass:** HTTP 200/201 — 64-character password accepted.  
**Fail:** HTTP 400 — application rejects passwords longer than ~20 chars (common framework misconfiguration).

---

### V2.1.3 (L1) — No password truncation

```bash
# Register with 72-char password, then log in with same password
# If the application truncates to bcrypt's 72-byte limit silently, login with the first 72 chars should still work
# but login with the full string should also work — both must succeed
```

**Pass:** Login succeeds with the full ≥ 64-character password.  
**Fail:** Login fails unless password is truncated, or application silently truncates on registration.

---

### V2.1.7 (L1) — Breached password check

```bash
# Attempt registration with a known-breached password
curl -s -X POST $TARGET/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test3@example.com","password":"Password1234"}' | jq '.error'
```

**Pass:** Application rejects the password with a message like "this password has appeared in a data breach."  
**Fail:** Account created with `Password1234` or any other top-10000 breached password.

---

### V2.2.1 (L1) — Anti-automation / lockout after failed attempts

```bash
for i in {1..6}; do
  CODE=$(curl -s -o /dev/null -w "%{http_code}" -X POST $TARGET/api/auth/login \
    -H "Content-Type: application/json" \
    -d '{"email":"victim@example.com","password":"wrongpass'$i'"}')
  echo "Attempt $i: $CODE"
done
```

**Pass:** Attempt 6 returns 429 (rate limited) or 423 (account locked).  
**Fail:** All attempts return 401 with no throttling or lockout applied.

---

### V2.2.3 (L1) — Credential recovery via secure channel only

```bash
# Trigger password reset
curl -s -X POST $TARGET/api/auth/forgot-password \
  -d '{"email":"victim@example.com"}'
# Check: does the response body contain the reset token/link directly?
# A pass means the token is only sent to the registered email
```

**Pass:** Response is generic ("if this email exists, a reset link was sent") — no token in response body.  
**Fail:** Reset token appears in the API response, or the reset link is predictable.

### V2.4.1 (L2) — Password hashing with approved algorithm

```bash
# Requires database/source access
# Option 1: source code review
grep -rn "bcrypt\|argon2\|scrypt\|pbkdf2\|Argon2" /app/src/ | grep -i "hash\|password"

# Option 2: timing oracle — bcrypt is slow (~100ms), MD5/SHA is fast (<1ms)
time curl -s -X POST $TARGET/api/auth/login \
  -d '{"email":"test@example.com","password":"ValidPass123!"}'
# bcrypt login takes 100–300ms; MD5/SHA login takes <5ms
```

**Pass:** Source confirms bcrypt/Argon2id/scrypt/PBKDF2 with appropriate work factor. Timing is consistent with slow hashing.  
**Fail:** Source shows MD5, SHA-1, SHA-256 without PBKDF2 iterations, or plaintext storage.

### V2.5.1 (L1) — No default credentials

```bash
# Common default credential pairs
DEFAULTS=("admin:admin" "admin:password" "root:root" "admin:123456" "test:test")

  echo "$cred -> $CODE"
done
```

**Pass:** All default credential attempts return 401.  
**Fail:** Any default pair returns 200 or a session cookie.

## V3 — Session Management

### V3.2.1 (L1) — New session token on authentication (session fixation)

```bash
# Step 1: get a pre-auth session cookie (e.g. from the login page)
PRE_AUTH=$(curl -sc /tmp/pre_cookies.txt $TARGET/login | grep -o 'session=[^;]*' | head -1)
echo "Pre-auth token: $PRE_AUTH"

# Step 2: authenticate
curl -sb /tmp/pre_cookies.txt -sc /tmp/post_cookies.txt \
  -X POST $TARGET/api/auth/login \
  -d '{"email":"test@example.com","password":"ValidPass123!"}'

POST_AUTH=$(grep session /tmp/post_cookies.txt | awk '{print $NF}')
echo "Post-auth token: $POST_AUTH"

[ "$PRE_AUTH" != "$POST_AUTH" ] && echo "PASS: token rotated" || echo "FAIL: same token — session fixation possible"
```

**Pass:** Post-authentication session token differs from pre-authentication token.  
**Fail:** Session token is the same before and after login.

---

### V3.2.3 (L1) — Secure and HttpOnly cookie flags

```bash
curl -sI -X POST $TARGET/api/auth/login \
  -d '{"email":"test@example.com","password":"ValidPass123!"}' | grep -i "set-cookie"
```

**Pass:** `Set-Cookie: session=...; HttpOnly; Secure; SameSite=Strict`  
**Fail:** Missing `HttpOnly` (JS-accessible), missing `Secure` (sent over HTTP), missing `SameSite` (CSRF risk).

---

### V3.3.1 (L1) — Logout invalidates server-side session

```bash
# Step 1: login and capture session token
TOKEN=$(curl -s -X POST $TARGET/api/auth/login \
  -d '{"email":"test@example.com","password":"ValidPass123!"}' | jq -r '.token')

# Step 2: confirm token works
curl -s -H "Authorization: Bearer $TOKEN" $TARGET/api/profile | jq '.email'

# Step 3: logout
curl -s -X POST $TARGET/api/auth/logout -H "Authorization: Bearer $TOKEN"

# Step 4: attempt to reuse the token
CODE=$(curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer $TOKEN" $TARGET/api/profile)
echo "Post-logout status: $CODE"
```

**Pass:** Step 4 returns 401.  
**Fail:** Step 4 returns 200 — server did not invalidate the token on logout.

---

### V3.3.2 (L2) — Idle session timeout ≤ 30 minutes

```bash
TOKEN=$(curl -s -X POST $TARGET/api/auth/login \
  -d '{"email":"test@example.com","password":"ValidPass123!"}' | jq -r '.token')

echo "Waiting 31 minutes..."
sleep 1860

CODE=$(curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer $TOKEN" $TARGET/api/profile)
echo "After 31min idle: $CODE"
```

**Pass:** Returns 401 after 30 minutes of inactivity.  
**Fail:** Returns 200 — session does not expire on idle.

---

### V3.4.1 (L1) — SameSite cookie attribute

```bash
curl -sI $TARGET | grep -i "set-cookie"
# Also check after login:
curl -sI -X POST $TARGET/api/auth/login -d '{"email":"test@example.com","password":"ValidPass123!"}' | grep -i "set-cookie"
```

**Pass:** `SameSite=Strict` or `SameSite=Lax` present on session cookie.  
**Fail:** `SameSite` attribute absent, or set to `None` without `Secure`.

---

### V3.5.1 (L2) — OAuth state parameter / CSRF token

```bash
# Initiate OAuth flow twice — state param must differ each time
curl -sI "$TARGET/oauth/authorize?response_type=code&client_id=app" | grep -i "location"
curl -sI "$TARGET/oauth/authorize?response_type=code&client_id=app" | grep -i "location"
# Both Location headers should contain unique ?state= values
```

**Pass:** Each authorization request uses a unique, unguessable state parameter.  
**Fail:** State parameter is absent, static, or predictable (sequential integer, timestamp).

---

### V3.7.1 (L2) — Re-authentication for sensitive actions

```bash
# After a long idle period or from a new IP, attempt a sensitive action (e.g. change email)
curl -s -X PUT $TARGET/api/account/email \
  -H "Authorization: Bearer $OLD_TOKEN" \
  -d '{"new_email":"attacker@evil.com"}'
```

**Pass:** Application requires current password confirmation or step-up MFA before email change.  
**Fail:** Email changed immediately with only the existing session token — no step-up auth required.

## V4 — Access Control

### V4.1.3 (L1) — Least privilege / horizontal privilege escalation (IDOR)

```bash
# Login as User A, get their resource ID
TOKEN_A="<user_a_jwt>"
RESOURCE_ID=$(curl -s -H "Authorization: Bearer $TOKEN_A" \
  $TARGET/api/documents | jq -r '.[0].id')

# Login as User B, attempt to access User A's resource
TOKEN_B="<user_b_jwt>"
CODE=$(curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer $TOKEN_B" \
  $TARGET/api/documents/$RESOURCE_ID)
echo "Cross-user access: $CODE"
```

**Pass:** Returns 403 Forbidden.  
**Fail:** Returns 200 with User A's data — IDOR vulnerability confirmed.

---

### V4.2.1 (L1) — IDOR on write operations (create/update/delete)

```bash
TOKEN_B="<user_b_jwt>"
RESOURCE_ID="<user_a_resource_id>"

# Attempt to delete User A's resource as User B
curl -s -o /dev/null -w "%{http_code}" -X DELETE \
  -H "Authorization: Bearer $TOKEN_B" \
  $TARGET/api/documents/$RESOURCE_ID

# Attempt to update
curl -s -o /dev/null -w "%{http_code}" -X PUT \
  -H "Authorization: Bearer $TOKEN_B" \
  $TARGET/api/documents/$RESOURCE_ID \
  -d '{"title":"hacked"}'
```

**Pass:** Both return 403.  
**Fail:** Delete or update succeeds — write-IDOR confirmed.

---

### V4.2.2 (L1) — Vertical privilege escalation (function-level access control)

```bash
TOKEN_USER="<standard_user_jwt>"

# Test admin endpoints with a regular user token
ADMIN_ENDPOINTS=(
  "/api/admin/users"
  "/api/admin/settings"
  "/api/admin/audit-log"
  "/api/internal/health"
)

for ep in "${ADMIN_ENDPOINTS[@]}"; do
  CODE=$(curl -s -o /dev/null -w "%{http_code}" \

```

**Pass:** All admin endpoints return 403 for a standard-user token.  
**Fail:** Any endpoint returns 200, 500 (unhandled — reveals functionality), or 404 only for unlisted endpoints while returning 200 for listed ones.

## V5 — Validation, Sanitization and Encoding

### V5.2.1 (L1) — XSS via HTML input

```bash
# Test stored XSS: submit a payload and check if it is reflected unescaped
XSS_PAYLOAD='<script>alert(document.domain)</script>'
curl -s -X POST $TARGET/api/comments \
  -H "Authorization: Bearer $TOKEN_A" \
  -H "Content-Type: application/json" \
  -d "{\"body\":\"$XSS_PAYLOAD\"}" > /dev/null

# Retrieve and check if payload is escaped
curl -s -H "Authorization: Bearer $TOKEN_A" $TARGET/api/comments | \
  grep -q "<script>" && echo "FAIL: XSS payload unescaped" || echo "PASS: payload escaped"
```

**Pass:** Stored payload is HTML-entity-encoded (`&lt;script&gt;`) in responses.  
**Fail:** Raw `<script>` tag appears in the response body.

---

### V5.3.4 (L1) — SQL injection prevention

```python
import requests

TARGET = "https://app.example.com"
PAYLOADS = [
    "' OR '1'='1",
    "'; DROP TABLE users;--",
    "1 UNION SELECT username,password FROM users--",
    "admin'--",
]

for payload in PAYLOADS:
    r = requests.get(f"{TARGET}/api/users", params={"id": payload}, timeout=10)
    if any(kw in r.text.lower() for kw in ["sql", "syntax", "mysql", "postgresql", "sqlite", "ora-"]):
        print(f"[FAIL] SQL error exposed for payload: {payload!r}")
    elif r.status_code == 200 and len(r.json()) > 1:
        print(f"[WARN] Possible SQLi — unexpected rows returned for: {payload!r}")
    else:
        print(f"[PASS] {payload!r} -> {r.status_code}")
```

**Pass:** All payloads return 400/422 validation error or a clean empty result with no SQL error strings.  
**Fail:** SQL error message in response body, or more rows returned than expected (union injection).

---

### V5.3.3 (L1) — Reflected XSS

```bash
# Check URL parameters and form fields for reflection
ENCODED="%3Cscript%3Ealert(1)%3C%2Fscript%3E"
curl -s "$TARGET/search?q=$ENCODED" | grep -q "<script>alert" && \
  echo "FAIL: reflected XSS" || echo "PASS: input encoded"
```

**Pass:** `<script>` tags in URL parameters are entity-encoded in the response.  
**Fail:** Raw script tag reflected — reflected XSS confirmed.

---

### V5.3.8 (L1) — OS command injection

```bash
PAYLOADS=("; ls -la" "| id" "\`whoami\`" "\$(id)" "; sleep 5")
for p in "${PAYLOADS[@]}"; do
  START=$(date +%s)
  curl -s -X POST $TARGET/api/convert \
    -d "filename=test.pdf$p" > /dev/null
  END=$(date +%s)
  ELAPSED=$((END - START))
  [ $ELAPSED -ge 5 ] && echo "FAIL: sleep injection succeeded ($p)" || echo "PASS: $p"
done
```

**Pass:** No command output in responses; sleep payload does not delay the response by ≥5 seconds.  
**Fail:** Command output appears in response, or sleep injection causes measurable delay (time-based blind injection).


### V6.2.2 (L2) — No custom/broken cryptography

```bash

grep -rn "crypto\|cipher\|encrypt\|decrypt" /app/src/ | \
```

**Pass:** No custom crypto routines found; TLS uses only AEAD ciphers (AES-GCM, ChaCha20-Poly1305).  
**Fail:** Custom XOR/ROT cipher found, or weak TLS ciphers negotiated.

```bash
  -d '{"to":"user_b","amount":-100}' | jq .

# Also test: amount=0, amount=99999999999, amount=0.001 (if integer-only expected)
```

**Pass:** Negative/zero/overflow amounts are rejected with validation error.  
**Fail:** Negative transfer processed — sender's balance increases (business logic flaw).

### V12.1.1 (L1) — File upload size limit enforced


### V12.3.1 (L1) — Path traversal prevention

```bash
TRAVERSALS=(
  "../../../../etc/passwd"
  "..%2F..%2F..%2Fetc%2Fpasswd"
  "....//....//etc/passwd"
  "%2e%2e%2f%2e%2e%2fetc%2fpasswd"
)
for t in "${TRAVERSALS[@]}"; do
  RESULT=$(curl -s "$TARGET/api/files?name=$t")
  echo "$RESULT" | grep -q "root:" && echo "FAIL: path traversal — /etc/passwd read" || echo "PASS: $t"
done
```

**Pass:** All traversal attempts return 400 or a generic file-not-found error.  
**Fail:** `/etc/passwd` content (or any file contents outside the intended directory) returned.

### V12.5.1 (L1) — Executable file upload blocked

## V13 — API and Web Service Verification

### V13.1.3 (L1) — No HTTP verbs that bypass access control

```bash
# Some frameworks treat OPTIONS, HEAD, TRACE differently — test all verbs on protected endpoints
for METHOD in GET POST PUT DELETE PATCH OPTIONS HEAD TRACE; do
  CODE=$(curl -s -o /dev/null -w "%{http_code}" -X $METHOD \
    $TARGET/api/admin/users)

```

**Pass:** All verbs return 403 (or 405 Method Not Allowed) for an unauthenticated request to a protected endpoint.  
**Fail:** OPTIONS or HEAD returns 200, bypassing the authorization check applied to GET/POST.
